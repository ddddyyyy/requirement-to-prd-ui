# Requirement to PRD & UI Prototype

## 核心功能

将用户需求对话转化为结构化的产品需求文档(PRD)和网页原型，支持从需求澄清、文档生成、自动自检到原型生成的完整工作流。

1. **完整PRD生成** - 根据用户输入，先进行需求澄清，然后生成包含需求背景、功能描述、用户故事、验收标准、原型图、UI设计稿等的完整PRD文档
2. **文档自检** - 自动检测文档中的冲突和不一致问题(术语一致性、逻辑一致性、UI和需求匹配度、依赖关系等)
3. **网页原型生成** - 根据UI设计稿生成可交互的HTML原型

## 快速开始

直接告诉我你的产品需求，比如：
```
我想做一个任务管理应用
```

## 可用命令

| 命令 | 描述 |
|------|------|
| `/prd-generate` | 生成或更新PRD文档(包含需求澄清) |
| `/prd-check` | 检查现有PRD文档的冲突和不一致性 |
| `/prototype-generate` | 根据PRD或UI设计稿生成网页原型 |
| `/full-process` | 执行完整流程：需求澄清 → PRD生成 → 文档自检 → 原型生成(每步需用户确认) |
| `/full-process --resume` | 从上次中断的位置继续执行 |
| `/full-process --restart` | 重新开始完整流程 |
| `/prd-status` | 查看当前项目进度状态 |

## 工作流程

本技能支持灵活的工作模式，可以从任意步骤开始：

1. **PRD生成阶段**
   - 需求澄清：通过结构化提问明确目标平台、技术栈、核心功能等
   - 生成PRD：输出包含需求背景、功能描述、用户故事、验收标准、Mermaid原型图、UI设计稿的完整文档
   - 参考规范：[PM.md](reference/PM.md)

2. **文档自检阶段**
   - 自动检测：术语一致性、逻辑一致性、UI与需求匹配度、依赖关系合理性
   - 问题确认：列出发现的问题，等待用户确认是否需要修改
   - 迭代优化：根据用户反馈修改文档，重新检查直至无冲突

3. **原型生成阶段**
   - 基于PRD中的UI设计稿或用户提供的设计稿
   - 生成可交互的HTML原型文件(包含CSS和JavaScript)
   - 参考规范：[UIPrototypeGen.md](reference/UIPrototypeGen.md)

## 进度反馈与中断恢复

**进度反馈机制：**
- 在执行 `/full-process` 或任何多步骤操作时，每完成一个阶段都会显示进度状态
- 进度格式：`[X/Y] 阶段名称 - 状态(进行中/已完成/待确认)`

**中断恢复机制：**
- 系统会自动保存每个阶段的完成状态到 `.prd-progress.json` 文件
- 如果会话中断，下次执行命令时会检测到未完成的进度
- 使用 `/full-process --resume` 或 `/full-process --continue` 可从上次中断的位置继续
- 使用 `/full-process --restart` 可重新开始完整流程
- 使用 `/prd-status` 可查看当前项目进度状态

**状态文件说明：**
- 位置：当前工作目录下的 `.prd-progress.json`
- 内容：包含各阶段完成状态、最后更新时间、PRD文件名等信息
- 手动删除该文件可重置所有进度

状态文件格式详细说明和进度管理规范参考 [PM.md](reference/PM.md)。

## 输出产物

- **PRD文档**：Markdown格式的标准产品需求文档，文件名格式：`00-项目需求文档.md`(数字递增避免重名)
- **网页原型**：独立的HTML文件，包含内嵌CSS/JS，可直接在浏览器中打开预览交互效果

## Gotchas

1. **必须先做需求澄清** - 不要跳过需求澄清直接生成PRD，除非用户明确提供了足够完整的需求信息
2. **确认门槛要明确** - 每个阶段完成后必须等待用户明确确认（如"继续"、"下一步"、"确认"）才能进入下一阶段，不要自动跳步
3. **状态文件必须及时更新** - 每个阶段完成后立即写入 `.prd-progress.json`，不要等到整个流程结束才保存
4. **避免内容重复** - PM.md和UIPrototypeGen.md中有详细规范，SKILL.md中只引用，不重复大段内容
5. **术语要统一** - 文档中同一概念必须使用相同词汇，发现不一致要在自检阶段指出
6. **UI设计稿要具体** - 不要只说"科技感蓝色"，要给出具体的十六进制色值、尺寸、间距等可落地的参数
7. **验收标准要可验证** - 必须采用"当[操作条件]时，系统应[预期结果]"格式，且包含可量化指标
8. **原型生成只还原不修改** - 严格按照PRD中的UI设计稿生成，不要随意修改设计元素

## 确认门槛

以下回复视为开始执行/继续的确认：
- "开始"、"确认"、"继续"、"下一步"、"好的"
- 明确的肯定性回复

以下回复不视为确认，需进一步澄清：
- "我看看"、"有道理"、"先这样"、"等一下"
- 模糊或中性的回复
